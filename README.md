# Excel FAQ RAG 客服机器人

这是一个可直接落地的版本：把 Excel 常见问题库作为唯一知识源，先做检索，再用 AI 生成更自然的客服话术。

现已支持 **三种工作模式**，可在客服工作台中随时切换：

| 模式 | 说明 |
| --- | --- |
| 全AI (`auto`) | AI 自动检索题库并直接回复客户，无需人工介入（原有行为，默认模式） |
| 全人工 (`manual`) | 所有问题进入人工队列，由客服手动查看并回复，AI 不参与 |
| 人机协同 (`collab`) | AI 先生成建议回复（仅客服可见），客服审核/编辑后再发送给客户；若超时客服还没有发送回复，系统会自动把 AI 建议发给客户 |

客户网页 (`/`) 始终只显示一个聊天框；如命中全人工或协同模式，客户会看到“正在转接人工客服”的提示并自动轮询等待回复，无需手动刷新。

人机协同模式下，AI 建议生成后会启动一个倒计时（默认 **5 秒**，可在客服工作台的“工作台设置”中调整，1-300 秒）：如果这段时间内客服还没有点击「发送回复」，系统会自动把 AI 建议的回复发给客户，避免客户长时间等待；只要客服在倒计时结束前发送了回复（无论是直接采纳 AI 建议还是自己改写），就不会再自动发送。

### 多产品支持

现已支持多个产品各自独立的题库（在 `web_app.py` 的 `PRODUCTS` 字典中配置）。客户在提问前需先在网页上选择要咨询的产品，选择结果会记录在浏览器本地并随每次提问一起提交；客服工作台的每条消息也会标注所属产品，方便区分处理。

**未命中题库的统一规则（任何模式下都适用）**：

- 若某个产品尚未配置 Excel 题库（`PRODUCTS` 中 `excel` 留空），该产品的所有提问都会被直接判定为"未命中"，不会消耗 AI 调用。
- 无论当前是全AI、全人工还是人机协同模式，只要题库检索未命中，系统都**不允许** AI 直接回复或编造答案，而是统一转人工处理（进入待处理队列）；只有确认命中题库时，全AI模式才会自动回复、协同模式才会生成 AI 建议并启动自动发送倒计时。
- **例外1**：单纯打招呼（"你好""在吗""hi"等、不带具体问题）不算未命中，任何模式下都会由 AI 直接回一句问候语，不转人工、不发邮件提醒；一旦打招呼的同时带了具体问题，仍会正常走题库检索流程。
- **例外2**：与产品咨询完全无关的闲聊/荒谬提问，或明显不构成真实客服需求的挑逗性/不当提问（"吃饭了嘛""这产品对国家安全有危害吗""哪里买到假货"之类；"哪里买到假货"特指字面上问"哪里能买到假货本身"，跟"怎么辨别真伪""在哪买正品才不会买到假货"这类正常防伪疑虑完全不同、后者仍会正常判定为相关；此外还包括内容本身低俗/色情/脏话骂人/人身攻击、明显违法违规、涉及政治敏感或国家安全违禁话题的提问，`_classify_irrelevant` 的 system prompt 里作为第 4 类单独列出）。这类说法千变万化，没法像打招呼一样靠关键词穷举，改用一次轻量 AI 判断：只有 AI **非常确信**完全无关/不构成真实需求时才直接回复引导语（第 4 类低俗/违法/涉政内容例外，只要命中就一律判定为无关，不会因为顺带提到产品就放过）；这类问题仍会完整落库（标记为 `matched=True`、`matched_question="无关闲聊/非常规提问"`，跟问候语的记录方式一致），方便管理员事后能在对话记录里查到，只是不进人工队列、不发邮件、不计入"转人工请求次数"统计；只要有一点点可能与产品相关，就判定为"相关"、正常走检索/转人工流程，避免真实问题被误判漏单。管理员可在工作台的「无关闲聊自动过滤」中随时关闭（对应 `skip_irrelevant_enabled` 设置，默认开启）。
  - **骂人/人身攻击内容的关键词兜底**：真实出现过"你是傻子"这种短句被 AI 误判成"命中题库、匹配度 100%"直接当正常问题回复的 bug（短文本语义检索容易凑巧跟无关的题库内容算出很高相似度，AI 语义判断对这种短句也不一定稳）。新增 `_is_abusive_language(text)`（正则匹配常见骂人词：傻子/笨蛋/蠢货/智障/脑残/白痴/废物等中英文粗话）和 `_is_irrelevant_question(question, product, model)`（先查关键词命中就直接判定无关，没命中再调 `_classify_irrelevant` 走 AI 语义判断）；`ask()` 和 `/transfer-question` 两处原来直接调 `_classify_irrelevant` 的地方都改成调 `_is_irrelevant_question`，保证骂人内容在进入题库检索之前就被拦下，不依赖 AI 调用是否成功。
- **例外3**：客户直接说"转人工""人工客服""找客服"等明确要求转接真人的话（不是一句需要检索/AI 判断的问题）。这类消息会在"无关闲聊"判断之前就被识别出来，跳过题库检索，直接进入转人工状态，不会先走一遍"AI 思考"的等待过程，也不会被无关闲聊过滤误判成"与产品无关"。因为"转人工"本身没有实际咨询内容，客户端会立即请客户补充一次具体问题（必填）+ 选填邮箱，详见下文"客户端等待提示"。
  - `/api/conversations/{id}/transfer-question` 接口对客户补充的这个具体问题，同样会调用 `_classify_irrelevant` 过一遍例外2那套判断（`skip_irrelevant_enabled` 开启时）——避免客户靠说一句"转人工"绕开题库检索这道过滤，把无关/低俗/违法/涉政内容硬推给人工客服。命中的话直接 `database.mark_answered(...)` 写入无关闲聊的引导语并返回 `{"success": true, "irrelevant": true, "answer": "..."}`，不再执行邮箱保存/邮件通知/`conversation_updated` 广播；前端 `renderTransferQuestionPrompt` 收到 `irrelevant: true` 后只展示这句引导语（`addMessage(data.answer, "bot")`）就直接 `return`，不展示邮箱气泡、不调用 `pollAnswer`，整个转人工流程到此终止。

### 转人工邮件提醒

一条提问进入人工队列时（题库未命中，或全人工模式下的任意问题），系统**不会**立即发邮件，只会把这条对话实时广播给客服工作台。邮件通知只有一个触发条件：**客户在等待过程中主动留下了合法格式的邮箱**（对应"留邮箱"或"补充问题+选填邮箱"这两个入口），此时才会真正发一封邮件通知客服/管理员，内容包含**客户（访客编号，跟工作台会话列表看到的编号一致）**、产品、模式、客户提问、客户邮箱、对话编号；客户不填邮箱则完全不会触发邮件。

- **全人工模式例外**：即使客户在该模式下留了邮箱，也**不会**发邮件——该模式下客服本就要一直盯着工作台处理每条对话，不需要邮件提醒；客户留的邮箱依然会保存、在工作台可见，方便客服直接联系，只是不触发邮件这个动作。

邮件通过标准 SMTP 发送，配置方式有两种，**后台配置优先于 `.env`**：

- **推荐：登录客服工作台配置**（「工作台设置 → SMTP 发信配置」）：管理员可以直接在页面上填发信服务器、端口、账号、密码、发信人、TLS/SSL，保存后立即生效，还能点「发送测试邮件」当场验证配置对不对，不需要 SSH 登录服务器、也不需要重启容器。密码字段留空保存表示不修改已保存的密码。配置存在数据库里（`data/qa.db`，已通过 Docker volume 持久化），换服务器部署也不会丢。
- **兜底：`.env` 里的 `SMTP_*` 变量**（见 `.env.example`）：后台还没配置过时（数据库里没有 `smtp_host`）会自动回退到这些环境变量；两者都没配置时会跳过发送并只打印警告日志，不影响客户提问、客服回复等主流程。

收件邮箱同理：默认取 `.env` 里的 `NOTIFY_EMAIL_TO`，管理员也可以在「工作台设置 → 邮件提醒收件人」里直接修改（支持逗号分隔多个收件人），后台配置优先、改动立即生效。

管理员还可以在客服工作台的「工作台设置」中额外开启**待处理队列定时提醒**：只要队列不为空，就按设定的分钟数（1-1440）周期性发一封汇总邮件，告知当前有多少位客户、多少条问题待处理，邮件里列出的每一条问题也会标注是哪个访客提出的；这个定时提醒与上面的即时提醒是两套独立机制，互不影响。

### 每日数据日报

管理员可在「工作台设置 → 每日数据日报」中开启/关闭，并设置每天的推送时间点（默认 **09:00，按香港时间计算**）。开启后，每天到点会自动发一封邮件，汇总**前一个香港日历日（00:00-24:00）**的数据：

- 咨询用户数（按客户去重）
- 总对话条数
- 转人工请求次数（口径：全人工模式下的任意问题，或全AI/协同模式下题库未命中——与是否实际发出邮件提醒无关）

发送渠道、收件邮箱复用上面的 SMTP 配置和邮件提醒收件人设置，不需要额外配置。

### 客户端等待提示

客户提问转人工后，网页先显示"已收到您的问题，正在为您转接人工客服，请稍候"，这条消息会**一直保留在对话记录里，不会被后续提示覆盖或删除**。

- **题库未命中（全AI / 协同模式）**：如果客服在 **10 秒**内还没有回复，会在这条消息**下方新增一条**消息，提示"人工客服正忙，留下你的邮箱，客服稍后通过邮箱回复您"，并就地提供邮箱输入框（对应后端 `/api/conversations/{id}/leave-email`）；客服最终的回复会写入这条新消息里。
- **全人工模式的任意问题**：**完全没有**上面这套"留邮箱"提示——问题已经直接进了工作台待处理队列，客服本就要一直盯着处理，客户端只会安静持续轮询，10 秒后不会弹出任何额外提示；客服最终在工作台回复后，答案会**另起一条新消息**展示，不会覆盖或替换掉"已收到您的问题…"这条记录。

如果客户是直接说"转人工"之类明确要求转接真人（见上面例外3），后端 `/api/ask` 会返回 `need_transfer_details: true`，客户端不需要等 10 秒、也不演"AI 思考"这段过程，立即展示**想咨询的问题**输入框（必填，提交后会替换掉数据库里"转人工"这句占位提问）（对应 `/api/conversations/{id}/transfer-question`）；若当前工作模式不是全人工模式，还会额外提供**邮箱**输入框（选填）——**全人工模式下不会展示这个邮箱输入框**，因为该模式完全不需要留邮箱这套机制。此时客户端还**不会**开始轮询（避免跟提交后另起的轮询互相冲突改同一条消息），要等客户真正提交这个表单之后才开始。

提交后，客户填的问题、邮箱（如果填了）都各自单独 `addMessage` 成一条正常大小的消息保留在对话记录里，不再是挤在提示语下面的一行 `.email-capture-status` 小字（那个小字样式现在只用来显示格式校验失败之类的即时错误提示）。紧接着新增一条 bot 消息作为轮询目标，文案按是否填了邮箱分两种：

- **没填邮箱**：显示「已将你的问题「XXX」更新给人工客服。」（`XXX` 是客户刚提交的具体问题），并以 `emailProvided: false` 调 `pollAnswer(conversationId, el, matched, mode, { question, emailProvided })`，跟题库未命中场景完全一样——等超过 10 秒客服还没回复，会在下面新增"人工客服正忙，留邮箱"提示，让客户这时候补填邮箱（提交成功后的确认文案同样带上客户具体问的内容和刚填的具体邮箱）。
- **填了邮箱**：**立即**显示「已将您的问题「XXX」更新给人工客服，请耐心等待客服回复您～」（带上客户具体问的内容），并以 `emailProvided: true`、`email: <客户填的邮箱>` 调 `pollAnswer`；如果客服在 10 秒内就回复了，直接替换掉这条确认消息；如果超过 10 秒仍未回复，会在下面**新增一条**静态确认消息「人工客服正忙，已邮件通知客服，请耐心等待客服通过邮箱【XXX@XXX.XXX】回复您～」（带上客户具体留的邮箱地址，不会再弹一次输入框追问邮箱，也不会重复"已将您的问题…更新给人工客服"这句已经说过的话）。所有"客服会通过邮箱回复"类的确认文案，邮箱地址都统一用「通过邮箱【具体地址】回复您」这种带书名号的格式，方便客户一眼认出哪部分是自己的邮箱。

`pollAnswer` 的 `options.question` 参数用来在"已将您的问题「XXX」更新给人工客服…"类确认文案里回显客户具体问的内容；`options.email` 用来在 10 秒后的确认文案里回显客户具体留的邮箱地址；`options.emailProvided` 决定 10 秒后到底是弹交互式的"留邮箱"输入框（`renderLeaveEmailPrompt`），还是直接换成一条带具体邮箱的静态确认消息。刷新页面恢复历史记录时（`restoreHistory`），后端 `/api/conversations/by-session/{session_id}` 会把客户自己留的邮箱原文一并返回（字段 `email`），确保刷新后这些确认文案里的邮箱地址依然能正确回显。

邮箱这一部分（题库未命中场景 / 非全人工模式下的补充问题场景）的规则都一样：后端先校验是否为**真实格式的邮箱**（形如 `xxx@xxx.xxx`，不是随便打几个字符），格式不对会提示错误并拒绝提交；只有格式合法才会真正落库并发一封邮件通知客服（内容含访客编号、客户提问、客户邮箱、对话编号），方便客服直接通过邮件回复。**转人工要真正通知客服，必须客户填写合法格式的邮箱**——客户如果不填邮箱，则不会触发这条邮件通知，但客服仍能在工作台的待处理队列里实时看到这条对话（问题内容的更新与邮箱是否填写无关，不填邮箱同样会更新问题内容）。**全人工模式下完全没有留邮箱这套机制**（无论是普通提问还是主动说"转人工"），客户端不会展示任何邮箱输入框，也不会有对应的邮件通知，因为该模式客服本就要一直盯着工作台。客户提交成功后，输入框位置会清空、由上面那条新增的 bot 消息承接后续状态。整个过程网页仍会在后台持续轮询（`pollAnswer` 不再设置最大轮询次数/超时兜底文案），一旦客服真正在网页端回复，答案会自动展示出来、无需客户手动刷新——AI 反馈不会超过 10 秒这个量级，涉及转人工的场景在 10 秒时已经进入上面的留邮箱/补充问题机制（全人工模式则是直接安静轮询），所以不需要再额外设置一个"等 10 分钟后放弃"的兜底提示。全人工模式下，客服的回复**永远另起一条新消息**展示，绝不会覆盖"已收到您的问题，正在为您转接人工客服，请稍候"这条记录；非全人工模式下如果客服在 10 秒内就回复了（还没弹出"留邮箱"提示或确认消息），回复会直接替换掉这条等待提示（相当于把"请稍候"自然过渡成真正的答案）。

### 客户端会话标识 / 刷新恢复聊天记录

`static/index.html` 用 `sessionStorage`（不是 `localStorage`）保存客户浏览器的会话标识 `fuyunhon_session_id`——`sessionStorage` 的生命周期跟"标签页"绑定：刷新页面（F5）不会清空，关闭标签页/浏览器窗口才会清空。借着这个特性：

- **刷新页面**：`sessionStorage` 里的会话 id 还在，页面加载时会调用 `GET /api/conversations/by-session/{session_id}`（只返回客户自己该看到的字段：问题、最终回复、状态、模式、题库命中情况、是否已留邮箱 `has_email`、邮箱原文 `email`、是否还在等待补充问题 `awaiting_transfer_details`、是否为主动转人工 `is_explicit_transfer`，不含 AI 建议草稿、题库匹配详情、客户 IP 等客服专用信息——客户自己留的邮箱原样返回给自己不算泄露）把历史问答重新渲染成消息气泡；如果最后一条问题还没有最终回复，会恢复出对应的等待提示文案并接回 `pollAnswer(..., { question, emailProvided: item.has_email, email: item.email })` 继续轮询，避免刷新后又重复弹一次"留邮箱"提示，也能在带具体邮箱的确认文案里正确回显。
- **关闭标签页/浏览器**：`sessionStorage` 被清空，下次打开是全新的会话 id，不会带着之前的聊天记录，工作台里也会被当成一个新的"访客编号"（`database.list_sessions` 按 `session_id` 首次提问时间分配编号）。
- **主动"转人工"但还没提交补充问题表单就刷新了页面**：这条对话的 `question` 字段目前还是"转人工"这句占位文本，如果按未命中题库的规则去恢复，会永久卡在一句不会再变的"已收到您的问题，正在为您转接人工客服，请稍候"，客户就没法再补充真正想问的内容了——这是刷新会改变客户端行为的一个真实 bug。修复方式：数据库给 `conversations` 表加了一列 `awaiting_transfer_details`（`is_explicit_transfer` 分支创建对话后置为 1；`/transfer-question` 提交时 `database.set_question` 会自动清成 0），`restoreHistory()` 读到这个字段为真时，不走 `pollAnswer`，而是重新调用 `renderTransferQuestionPrompt(...)`，弹出跟第一次一模一样的"请描述您的问题"输入框，让客户能接着把流程走完。
- **一条对话已经等了超过 10 秒（该出现"人工客服正忙"提示了）时刷新页面**：修复前 `pollAnswer` 的 `busyTimer` 一律固定等 10000ms，每次刷新都是从 0 重新数，导致客户明明已经等过、该看到的提示却像是"倒退"了一样要重新等一遍才出现。现在 `/api/conversations/by-session/{session_id}` 额外返回 `created_at`（服务端记录的真实提问时间，UTC），`restoreHistory()` 用 `elapsedMsSince(item.created_at)` 算出这条对话已经等了多久，传给 `pollAnswer(..., { elapsedMs })`；`pollAnswer` 内部把固定的 `10000` 改成 `remainingMs = Math.max(0, 10000 - elapsedMs)`——已经等超过 10 秒的会直接展示对应提示（`setTimeout(..., 0)` 相当于立即执行），还没到 10 秒的只等剩余时间，不会重新从 10 秒开始数。全新提问（未传 `elapsedMs`）时 `elapsedMs` 默认 0，行为跟之前完全一样。
- **刷新时同时恢复多条待处理对话，各自的"人工客服正忙"提示挤在一起、跟自己的提问脱节**：根因是 `pollAnswer` 的 `busyTimer` 触发后一直用 `addMessage(...)`（无条件 `appendChild` 到 `#messages` 容器最末尾）来插入新的提示气泡；刷新恢复历史时，多条对话的 `remainingMs` 大多是 0（早就该出现提示了），它们的 `setTimeout` 几乎在同一时刻触发，而此时 `forEach` 循环已经把全部对话的"问题 + 已收到您的问题"气泡都同步添加完了——所以这些异步补上的提示会全部堆在最后一条气泡后面，跟各自真正对应的提问完全脱节，看起来次序很乱。修复：新增 `addMessageAfter(afterEl, text, role)`，用 `afterEl.insertAdjacentElement("afterend", el)` 把新气泡固定插在指定元素后面而不是容器末尾；`pollAnswer` 里所有原来 `addMessage("", "bot")` 创建 `followUpEl`/客服回复气泡的地方都改成 `addMessageAfter(loadingEl, "", "bot")`，保证不管有多少条对话同时到期，各自的提示都紧跟在自己那条"已收到您的问题"气泡后面。
- **客户之前留过的邮箱，刷新页面后气泡"消失了"**：实时聊天时客户留的邮箱会单独显示成一条 `📧 邮箱地址` 消息，但 `restoreHistory()` 只重建了 `item.question` 对应的气泡，没有重建这条邮箱气泡（后端一直有正常保存并通过 `item.email` 返回，只是前端没画出来）。修复：`restoreHistory()` 在添加问题气泡后，若 `item.email` 存在则紧接着补一条 `📧 ${item.email}` 的客户气泡，跟实时聊天时的展示保持一致。
- **主动"转人工"补充问题后，刷新页面确认文案变成了另一套话术**：客户主动说"转人工"并补充具体问题后，实时看到的确认语是"已将您的问题「具体问题」更新给人工客服…"；但 `restoreHistory()` 之前无法区分"这条对话是主动转人工，还是题库未命中被动转人工"，一律恢复成被动转人工的话术"已收到您的问题，正在为您转接人工客服，请稍候"，跟客户当时实际看到的不一样。修复：数据库给 `conversations` 表新增一列 `is_explicit_transfer`（`INTEGER DEFAULT 0`）——跟 `awaiting_transfer_details` 不同，客户提交补充问题后这个标记**不会**被清零，永久记住这条对话最初的来源；`is_explicit_transfer` 分支创建对话时置为 1（`database.set_is_explicit_transfer`），`/by-session` 接口把它一并返回给前端。`restoreHistory()` 现在按 `item.is_explicit_transfer` 选择正确的确认语：为真时，按 `item.email` 是否存在还原出`已将您的问题「X」更新给人工客服，请耐心等待客服回复您～`（已留邮箱）或 `已将你的问题「X」更新给人工客服。`（未留邮箱）；为假时才恢复成原来的"已收到您的问题…"话术，跟被动转人工场景保持一致。

无关闲聊/荒谬提问（`_classify_irrelevant` 命中的情况）现在也会完整落库、正常返回真实的 `conversation_id`，因此刷新页面时同样会出现在恢复的历史记录里，跟其他正常问答一样。

### 客服工作台：回复框显示规则 / 草稿保存

回复框（`compose-area`）只在对话真正需要客服手动处理时才会渲染：题库未命中、全人工模式的任意问题、或客户主动要求转人工，这三种情况会显示；全AI模式已直接回复、或人机协同模式题库命中且 AI 已生成建议（进入自动发送倒计时）这两种情况**不会**显示回复框，因为这类对话不需要人工介入（AI 已经处理完，或即将自动发送）。

回复框只有一个「发送回复」按钮，客服可以随时直接输入内容并发送给客户；发送成功后对话变为"已回复"状态，回复框自动消失。（曾经有一套"认领此问题 / 释放认领"的多客服协作机制，只有多个客服同时在线可能抢同一个对话时才有意义，已去掉，`/api/agent/claim`、`/api/agent/release` 这两个接口以及对应的 `claimed` 对话状态不再产生新数据，`database.py` 里仍保留 `claimed_by` 字段和对 `'claimed'` 历史状态值的兼容处理，避免影响旧数据。）

由于工作台会定期（WebSocket 推送 + 5 秒兜底轮询）刷新整个消息列表，`static/agent.html` 用一个按对话 id 索引的 `composeDrafts` 内存对象缓存客服正在输入但还没发送的草稿：`textarea` 的 `input` 事件把内容写入草稿箱，重新渲染时优先读草稿箱、而不是每次都用 AI 建议覆盖，避免客服打字打到一半被刷新清空；发送成功或对话变为已回复后清除对应草稿。

草稿箱只能保住"已经敲下确认的字"，保不住中文输入法里还没上屏的拼音候选词——那部分状态是绑定在具体的 `textarea` DOM 节点上的，如果这时候整段重渲染（`renderThread` 每次都会 `innerHTML` 整个替换掉消息列表），候选词所在的节点被销毁重建，候选词就直接丢了。所以 `renderThread()` 开头会判断 `document.activeElement` 是不是当前正聚焦在某个 `[data-answer-input]` 输入框上，如果是就直接跳过这次渲染（不触碰 DOM）；`textarea` 的 `blur` 事件会补触发一次 `renderThread()`，把输入期间攒下的更新追平。

- **右上角模式快捷切换**（仅管理员可点击）：点击即可在全AI / 全人工 / 人机协同之间切换，立即生效，无需重启服务；非管理员只会看到当前模式的只读提示。
- **工作台设置**（右上角按钮，弹出面板）：管理员可在此配置 SMTP 发信服务器（含一键测试发送）、邮件提醒收件人、调整人机协同的自动发送超时时间、开启/关闭待处理队列定时提醒邮件并设置提醒间隔（分钟）、开启/关闭每日数据日报并设置推送时间点（香港时间），以及管理客服账号。
- **待处理队列**：全人工/协同模式下的新问题会实时推送到所有在线客服（WebSocket + 5 秒兜底轮询），支持多人同时在线。
  - 协同模式下，队列条目会显示 AI 生成的建议回复（含匹配度），客服可直接采纳或编辑后发送。
  - 回复框只有一个「发送回复」按钮，任意在线客服都可以直接输入内容并发送，不需要先"认领"。
- **会话列表搜索 + 未处理红点**：会话列表顶部有一个搜索框，客户没有真实姓名，所以按"访客编号"匹配（输入 `13` 或 `访客13` 均可），跟"全部/待处理/已回复"筛选可以同时生效（`static/agent.html` 里的 `matchesSearch()`）；每个客户头像右上角，只要该客户还有 `pending_count > 0` 的未处理问题，就会显示一个红色小圆点（`.unread-dot`），客服扫一眼列表就能看出哪些客户还没处理完，不需要逐条点开确认。
  - 会话列表每一行不再显示"产品"徽章：因为徽章之前取的是 `session.last_product`（该客户最后一条消息咨询的产品），而同一个客户完全可能先后咨询过多个不同产品，只显示最后一个会误导客服。产品信息仍保留在点开会话后每条具体消息自己的徽章上（`item.product`，每条消息各自记录当时提问的产品，不会混淆）。
- **历史记录**：展示最近 50 条问题（覆盖全AI、全人工、人机协同三种模式），每条都会标注「题库检索标注」——命中了题库中的哪个问题及对应答案，或提示未命中，方便复核 AI 是否按题库准确回复。
- **客服账号管理**（仅管理员）：可在工作台内直接创建新的客服/管理员账号。

> 建议登录后立即修改默认管理员密码：可调用 `POST /api/agent/change-password`（`{"old_password":"...","new_password":"..."}`，需携带登录返回的 `token`）。

## 6) 参数说明

**回答机制**：每次提问分别用关键词检索（TF-IDF）和语义向量检索（embedding 余弦相似度）
各取 `top-k` 条候选，取并集后交给 AI 在候选中做语义判断——即使客户的措辞、语序、同义/口语化
表达（例如"涨豆豆"对应"长痘"）与题库原文不同，只要意图相同即可命中；语义检索弥补了纯关键词
检索对这类改写问句召回不足的问题。命中后，AI 只生成一句开场白/过渡语，专业内容（成分、功效、
用法用量、禁忌等）始终是题库答案原文的直接拼接，不经过 AI 改写，确保逐字一致。如果 AI 判断
所有候选都不匹配，或调用失败兜底走关键词分数判断，会返回引导客户补充关键词的话术。每次提问的
检索结果（命中的题库问题/答案，或未命中）都会记录在后台，客服工作台的问题卡片上会显示
「题库检索标注」供人工核对。首次启动时会调用一次 embedding 接口为所有题库问题建立语义索引，
如果该接口调用失败（无网络/无 API Key），会自动降级为纯关键词检索，不影响其余功能。

- `--top-k 3`：每次返回的知识条数，默认 3
- `--min-score 0.1`：最低命中阈值，太低就会拒答并引导补充问题
- `--self-test`：跑一次样例问答后退出，便于快速验收

网页版本也支持这些环境变量：

- `FAQ_EXCEL_PATH`：Excel 路径，默认读取当前目录的肤润康题库
- `FAQ_TOP_K`：关键词检索和语义检索各自的候选条数（取并集后交给 AI 做语义匹配），默认 8
- `FAQ_MIN_SCORE`：AI 语义匹配调用失败时，兜底关键词检索所用的最低命中阈值，默认 0.1
- `FAQ_EMBED_MODEL`：语义检索所用的 embedding 模型，默认 `text-embedding-v3`
- `OPENAI_MODEL`：模型名，默认 `qwen3.6-flash`
- `OPENAI_BASE_URL`：默认 `https://dashscope-intl.aliyuncs.com/compatible-mode/v1`

## 7) GitHub 自动部署到阿里云 ECS

仓库已包含 `.github/workflows/deploy.yml`。

在 GitHub → Settings → Secrets and variables → Actions 添加：

- `ECS_HOST`：服务器公网 IP
- `ECS_USER`：SSH 用户名（如 `root`）
- `ECS_PASSWORD`：SSH 密码

服务器需先手动部署一次，并保留 `~/fuyunhon_cs/.env`（不要提交到 Git）。

之后每次 `push` 到 `main`，GitHub Actions 会自动：

1. SSH 登录 ECS
2. `git pull origin main`
3. `docker compose up -d --build`

> **升级到三种模式后的一次性操作**：由于 `.env` 不会被 `git pull` 更新，首次部署本次更新前，请手动 SSH 登录服务器，在 `~/fuyunhon_cs/.env` 中补充 `JWT_SECRET`、`ADMIN_USERNAME`、`ADMIN_PASSWORD`、`DEFAULT_MODE`（参考 `.env.example`），保存后再让 GitHub Actions 触发部署，或手动执行一次 `docker compose up -d --build`。同时确保服务器上存在 `~/fuyunhon_cs/data` 目录（`docker-compose.yml` 已配置挂载，用于持久化客服账号与对话数据）。
