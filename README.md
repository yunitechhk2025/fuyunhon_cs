# Excel FAQ RAG 客服机器人

这是一个可直接落地的版本：把 Excel 常见问题库作为唯一知识源，先做检索，再用 AI 生成更自然的客服话术。

现已支持 **三种工作模式**，可在客服工作台中随时切换：

| 模式 | 说明 |
| --- | --- |
| 全AI (`auto`) | AI 自动检索题库并直接回复客户，无需人工介入（原有行为，默认模式） |
| 全人工 (`manual`) | 所有问题进入人工队列，由客服手动查看并回复，AI 不参与 |
| 人机协同 (`collab`) | AI 先生成建议回复（仅客服可见），客服审核/编辑后再发送给客户；若超时未被认领，系统会自动把 AI 建议发给客户 |

客户网页 (`/`) 始终只显示一个聊天框；如命中全人工或协同模式，客户会看到“正在转接人工客服”的提示并自动轮询等待回复，无需手动刷新。

人机协同模式下，AI 建议生成后会启动一个倒计时（默认 **5 秒**，可在客服工作台的“工作台设置”中调整，1-300 秒）：如果这段时间内没有客服认领该问题，系统会自动把 AI 建议的回复发给客户，避免客户长时间等待；一旦客服认领，则完全交由人工处理，不会再自动发送。

### 多产品支持

现已支持多个产品各自独立的题库（在 `web_app.py` 的 `PRODUCTS` 字典中配置）。客户在提问前需先在网页上选择要咨询的产品，选择结果会记录在浏览器本地并随每次提问一起提交；客服工作台的每条消息也会标注所属产品，方便区分处理。

**未命中题库的统一规则（任何模式下都适用）**：

- 若某个产品尚未配置 Excel 题库（`PRODUCTS` 中 `excel` 留空），该产品的所有提问都会被直接判定为"未命中"，不会消耗 AI 调用。
- 无论当前是全AI、全人工还是人机协同模式，只要题库检索未命中，系统都**不允许** AI 直接回复或编造答案，而是统一转人工处理（进入待处理队列）；只有确认命中题库时，全AI模式才会自动回复、协同模式才会生成 AI 建议并启动自动发送倒计时。
- **例外1**：单纯打招呼（"你好""在吗""hi"等、不带具体问题）不算未命中，任何模式下都会由 AI 直接回一句问候语，不转人工、不发邮件提醒；一旦打招呼的同时带了具体问题，仍会正常走题库检索流程。
- **例外2**：与产品咨询完全无关的闲聊/荒谬提问，或明显不构成真实客服需求的挑逗性提问（"吃饭了嘛""这产品对国家安全有危害吗""哪里买到假货"之类，后者特指字面上问"哪里能买到假货本身"，跟"怎么辨别真伪""在哪买正品才不会买到假货"这类正常防伪疑虑完全不同、后者仍会正常判定为相关）。这类说法千变万化，没法像打招呼一样靠关键词穷举，改用一次轻量 AI 判断：只有 AI **非常确信**完全无关/不构成真实需求时才直接回复引导语，不落库、不进队列、不发邮件；只要有一点点可能与产品相关，就判定为"相关"、正常走检索/转人工流程，避免真实问题被误判漏单。管理员可在工作台的「无关闲聊自动过滤」中随时关闭（对应 `skip_irrelevant_enabled` 设置，默认开启）。
- **例外3**：客户直接说"转人工""人工客服""找客服"等明确要求转接真人的话（不是一句需要检索/AI 判断的问题）。这类消息会在"无关闲聊"判断之前就被识别出来，跳过题库检索，直接进入转人工状态，不会先走一遍"AI 思考"的等待过程，也不会被无关闲聊过滤误判成"与产品无关"。因为"转人工"本身没有实际咨询内容，客户端会立即请客户补充一次具体问题（必填）+ 选填邮箱，详见下文"客户端等待提示"。

### 转人工邮件提醒

只要一条提问需要人工处理，系统就会立即发一封邮件提醒客服/管理员，邮件内容包含产品、模式、转人工原因、以及客户的原始提问：

- **全AI / 人机协同模式**：仅当题库检索**未命中**时才发邮件（命中时 AI 已直接回复或已生成建议，无需打扰人工）。
- **全人工模式**：因为该模式下所有问题都不会由 AI 直接作答，所以**每一条新问题**都会发邮件提醒。
- **例外**：客户主动说"转人工"（上面的例外3）不适用"立即发邮件"——此时还没有实际问题内容，发邮件对客服没意义，要等客户提交问题后才会广播给工作台，只有同时留了合法邮箱才会额外发邮件通知（见下文"客户端等待提示"）。

邮件通过标准 SMTP 发送，配置方式有两种，**后台配置优先于 `.env`**：

- **推荐：登录客服工作台配置**（「工作台设置 → SMTP 发信配置」）：管理员可以直接在页面上填发信服务器、端口、账号、密码、发信人、TLS/SSL，保存后立即生效，还能点「发送测试邮件」当场验证配置对不对，不需要 SSH 登录服务器、也不需要重启容器。密码字段留空保存表示不修改已保存的密码。配置存在数据库里（`data/qa.db`，已通过 Docker volume 持久化），换服务器部署也不会丢。
- **兜底：`.env` 里的 `SMTP_*` 变量**（见 `.env.example`）：后台还没配置过时（数据库里没有 `smtp_host`）会自动回退到这些环境变量；两者都没配置时会跳过发送并只打印警告日志，不影响客户提问、客服回复等主流程。

收件邮箱同理：默认取 `.env` 里的 `NOTIFY_EMAIL_TO`，管理员也可以在「工作台设置 → 邮件提醒收件人」里直接修改（支持逗号分隔多个收件人），后台配置优先、改动立即生效。

管理员还可以在客服工作台的「工作台设置」中额外开启**待处理队列定时提醒**：只要队列不为空，就按设定的分钟数（1-1440）周期性发一封汇总邮件，告知当前有多少位客户、多少条问题待处理；这个定时提醒与上面的即时提醒是两套独立机制，互不影响。

### 每日数据日报

管理员可在「工作台设置 → 每日数据日报」中开启/关闭，并设置每天的推送时间点（默认 **09:00，按香港时间计算**）。开启后，每天到点会自动发一封邮件，汇总**前一个香港日历日（00:00-24:00）**的数据：

- 咨询用户数（按客户去重）
- 总对话条数
- 转人工请求次数（口径与上面的"转人工邮件提醒"一致：全人工模式下的任意问题，或全AI/协同模式下题库未命中）

发送渠道、收件邮箱复用上面的 SMTP 配置和邮件提醒收件人设置，不需要额外配置。

### 客户端等待提示

客户提问转人工后，如果客服在 **10 秒**内还没有回复，网页会提示"人工客服正忙，留下你的電郵，客服稍後通過電郵回復您"，并在下方就地提供邮箱输入框（对应后端 `/api/conversations/{id}/leave-email`）。

如果客户是直接说"转人工"之类明确要求转接真人（见上面例外3），后端 `/api/ask` 会返回 `need_transfer_details: true`，客户端不需要等 10 秒、也不演"AI 思考"这段过程，立即展示两个输入框（对应 `/api/conversations/{id}/transfer-question`）：**想咨询的问题**（必填，提交后会替换掉数据库里"转人工"这句占位提问）+ **邮箱**（选填）。

无论走哪条路径，邮箱这一部分的规则都一样：后端先校验是否为**真实格式的邮箱**（形如 `xxx@xxx.xxx`，不是随便打几个字符），格式不对会提示错误并拒绝提交；只有格式合法才会真正落库并发一封邮件通知客服（内容含客户提问、客户邮箱、对话编号），方便客服直接通过邮件回复。**转人工要真正通知客服，必须客户填写合法格式的邮箱**——客户如果不填邮箱，则不会触发这条邮件通知，但客服仍能在工作台的待处理队列里实时看到这条对话（问题内容的更新与邮箱是否填写无关，不填邮箱同样会更新问题内容）。客户提交成功后会看到"已将您的问题更新给人工客服"的确认提示。整个过程网页仍会在后台继续轮询，一旦客服真正在网页端回复，会自动替换成实际内容，无需客户手动刷新。

### 客服工作台：回复框显示规则 / 认领机制 / 草稿保存

回复框（`compose-area`）只在对话真正需要客服手动处理时才会渲染：题库未命中、全人工模式的任意问题、或客户主动要求转人工，这三种情况会显示；全AI模式已直接回复、或人机协同模式题库命中且 AI 已生成建议（进入自动发送倒计时）这两种情况**不会**显示回复框，因为这类对话不需要人工介入（AI 已经处理完，或即将自动发送）。

「认领此问题」/「释放认领」只是客服之间的协作提示，不是回复客户的前提条件——不认领也能直接在回复框输入内容并点「发送回复」。认领后其他客服打开该对话会看到"该客服正在处理"的提示，避免重复答复；释放认领可以让其他客服接手。回复发送成功后对话变为"已回复"状态，回复框和认领按钮一起消失。

由于工作台会定期（WebSocket 推送 + 5 秒兜底轮询）刷新整个消息列表，`static/agent.html` 用一个按对话 id 索引的 `composeDrafts` 内存对象缓存客服正在输入但还没发送的草稿：`textarea` 的 `input` 事件把内容写入草稿箱，重新渲染时优先读草稿箱、而不是每次都用 AI 建议覆盖，避免客服打字打到一半被刷新清空；发送成功或对话变为已回复后清除对应草稿。

## 1) 准备 Excel 题库

把你的 Excel 文件放到当前目录，例如：`客服知识库.xlsx`。

支持 `.xls` / `.xlsx`，支持多 sheet。

当前脚本优先识别包含以下字段的 FAQ 表头：

- `咨询问题`
- `解答`
- （可选）`序号` / `编号`

当前 `docker-compose.yml` 默认挂载这个文件：

```text
2026.01.26_肤润康-常见咨询问题_v2(1).xls
```

如果你更换文件名，需要同步修改 `docker-compose.yml` 里的 `volumes`。

## 2) 配置 .env

打开 `.env`，把第一行替换成真实 Key：

```env
OPENAI_API_KEY=你的真实key
```

如果你是兼容接口（例如本地网关），再加：

```env
OPENAI_BASE_URL=https://你的兼容接口/v1
```

常用配置：

```env
OPENAI_MODEL=gpt-4o-mini
FAQ_TOP_K=8
FAQ_MIN_SCORE=0.1
APP_PORT=8000
```

客服工作台相关配置（新增）：

```env
# JWT 签名密钥，务必修改为随机字符串
JWT_SECRET=change_this_to_a_random_secret_string

# 首次启动自动创建的管理员账号，登录后请立即修改密码
ADMIN_USERNAME=admin
ADMIN_PASSWORD=change_this_password

# 系统首次启动时的默认模式：auto / manual / collab
DEFAULT_MODE=auto
```

> 首次启动会自动在数据库里创建管理员账号（用户名/密码取自上面的环境变量），日志里也会打印一次。数据库文件保存在 `data/qa.db`，`docker-compose.yml` 已挂载该目录为宿主机卷，重启/重新构建容器不会丢失客服账号、模式设置与历史对话。

## 4) 客服工作台

访问：

```text
http://127.0.0.1:8000/agent
```

用 `.env` 中配置的管理员账号登录（默认 `admin` / 见 `ADMIN_PASSWORD`）。

- **右上角模式快捷切换**（仅管理员可点击）：点击即可在全AI / 全人工 / 人机协同之间切换，立即生效，无需重启服务；非管理员只会看到当前模式的只读提示。
- **工作台设置**（右上角按钮，弹出面板）：管理员可在此配置 SMTP 发信服务器（含一键测试发送）、邮件提醒收件人、调整人机协同的自动发送超时时间、开启/关闭待处理队列定时提醒邮件并设置提醒间隔（分钟）、开启/关闭每日数据日报并设置推送时间点（香港时间），以及管理客服账号。
- **待处理队列**：全人工/协同模式下的新问题会实时推送到所有在线客服（WebSocket + 5 秒兜底轮询），支持多人同时在线。
  - 协同模式下，队列条目会显示 AI 生成的建议回复（含匹配度），客服可直接采纳或编辑后发送。
  - 点击「认领此问题」后，该问题会标记为你在处理，其他客服会看到「客服 X 正在处理中」，避免重复回复。
  - 「释放认领」可把问题放回待处理队列，交给其他客服处理。
- **历史记录**：展示最近 50 条问题（覆盖全AI、全人工、人机协同三种模式），每条都会标注「题库检索标注」——命中了题库中的哪个问题及对应答案，或提示未命中，方便复核 AI 是否按题库准确回复。
- **客服账号管理**（仅管理员）：可在工作台内直接创建新的客服/管理员账号。

> 建议登录后立即修改默认管理员密码：可调用 `POST /api/agent/change-password`（`{"old_password":"...","new_password":"..."}`，需携带登录返回的 `token`）。

## 6) 参数说明

**回答机制**：每次提问分别用关键词检索（TF-IDF）和语义向量检索（embedding 余弦相似度）
各取 `top-k` 条候选，取并集后交给 AI 在候选中做语义判断——即使客户的措辞、语序、同义/口语化
表达（例如"涨豆豆"对应"长痘"）与题库原文不同，只要意图相同即可命中；语义检索弥补了纯关键词
检索对这类改写问句召回不足的问题。命中后，AI 只生成一句开场白/过渡语，专业内容（成分、功效、
用法用量、禁忌等）始终是题库答案原文的直接拼接，不经过 AI 改写，确保逐字一致。如果 AI 判断
所有候选都不匹配，或调用失败兜底走关键词分数判断，会返回引导客户补充关键词的话术。每次提问的
检索结果（命中的题库问题/答案，或未命中）都会记录在后台，客服工作台的问题卡片上会显示
「题库检索标注」供人工核对。首次启动时会调用一次 embedding 接口为所有题库问题建立语义索引，
如果该接口调用失败（无网络/无 API Key），会自动降级为纯关键词检索，不影响其余功能。

- `--top-k 3`：每次返回的知识条数，默认 3
- `--min-score 0.1`：最低命中阈值，太低就会拒答并引导补充问题
- `--self-test`：跑一次样例问答后退出，便于快速验收

网页版本也支持这些环境变量：

- `FAQ_EXCEL_PATH`：Excel 路径，默认读取当前目录的肤润康题库
- `FAQ_TOP_K`：关键词检索和语义检索各自的候选条数（取并集后交给 AI 做语义匹配），默认 8
- `FAQ_MIN_SCORE`：AI 语义匹配调用失败时，兜底关键词检索所用的最低命中阈值，默认 0.1
- `FAQ_EMBED_MODEL`：语义检索所用的 embedding 模型，默认 `text-embedding-v3`
- `OPENAI_MODEL`：模型名，默认 `qwen3.6-flash`
- `OPENAI_BASE_URL`：默认 `https://dashscope-intl.aliyuncs.com/compatible-mode/v1`

## 7) GitHub 自动部署到阿里云 ECS

仓库已包含 `.github/workflows/deploy.yml`。

在 GitHub → Settings → Secrets and variables → Actions 添加：

- `ECS_HOST`：服务器公网 IP
- `ECS_USER`：SSH 用户名（如 `root`）
- `ECS_PASSWORD`：SSH 密码

服务器需先手动部署一次，并保留 `~/fuyunhon_cs/.env`（不要提交到 Git）。

之后每次 `push` 到 `main`，GitHub Actions 会自动：

1. SSH 登录 ECS
2. `git pull origin main`
3. `docker compose up -d --build`

> **升级到三种模式后的一次性操作**：由于 `.env` 不会被 `git pull` 更新，首次部署本次更新前，请手动 SSH 登录服务器，在 `~/fuyunhon_cs/.env` 中补充 `JWT_SECRET`、`ADMIN_USERNAME`、`ADMIN_PASSWORD`、`DEFAULT_MODE`（参考 `.env.example`），保存后再让 GitHub Actions 触发部署，或手动执行一次 `docker compose up -d --build`。同时确保服务器上存在 `~/fuyunhon_cs/data` 目录（`docker-compose.yml` 已配置挂载，用于持久化客服账号与对话数据）。
